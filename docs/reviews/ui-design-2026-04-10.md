## What Works Well (Top 3 Wins)

### 1. Mobile-First Layout Architecture

The `max-width: 430px` centered container with `100dvh` height, flex column structure (ResourceBar -> screenArea -> NavBar), and touch-optimized interactions is excellent. This is the correct foundation for a game that will live on mobile and Steam (via Electron/Steamworks overlay). The `-webkit-overflow-scrolling: touch`, `-webkit-tap-highlight-color: transparent`, safe-area-inset handling, and 4px custom scrollbar all demonstrate real mobile craft. This architecture should NOT change during the pixel art transformation.

### 2. Progressive Disclosure Through Level Unlocks

The NavBar's locked tab system (showing "Lv.3", "Lv.5", etc. with lock icons and 35% opacity) is a textbook implementation of progressive complexity. New players see 2 tabs (Hub + Forge). Content reveals itself as the player levels. The Hub's nav tiles mirror this with unlock indicators. This reduces cognitive load for new players while creating clear goals. The "Next Goal" card on Hub reinforces this by always pointing the player toward the next unlock. This is Duolingo-level onboarding thinking.

### 3. Forgelight Color System Coherence

The palette is consistent across all screens. Each screen has a signature accent color (Hub: orange, Forge: orange, Barracks: blue, World Map: green, Season: purple/seasonal, Village: amber). Resources always appear with their canonical colors. The dark background (#07070f) with rgba(255,255,255,0.03) surface cards creates depth without competing with content. This coherence would survive a pixel art transformation intact -- it is the emotional skeleton of the game.

## What Needs Improvement (Top 3 Issues)

### 1. CRITICAL: Every Surface Reads as "SaaS Dashboard," Not "Game"

**The Problem:**
Every card, panel, and container in the game uses the same visual formula: `background: rgba(255,255,255,0.03)` + `border: 1px solid rgba(255,255,255,0.06)` + `border-radius: 12-14px`. This is the visual language of Linear, Vercel, or any dark-mode SaaS app. It is competent CSS. It is also completely genre-inappropriate for a medieval fantasy RPG.

The Hub screen alone has 7+ cards using this exact pattern (playerCard, goalCard, chestsSection, tile, heroSummary, expeditionBanner, notifications). The Forge has the same for queue, recipeCard, inventoryGrid items. The Barracks detail panel, stat cells, equip slots, endurance section -- all identical glass-morphism rectangles.

**Why It Matters:**
Steam players browse by thumbnail and capsule image. A screenshot of Forge & Field at 300px wide is indistinguishable from a crypto wallet app or a project management tool. There is no visual signal that says "this is a game." The rounded-rectangle card is the most overused UI primitive in 2025 web design. It communicates "utility" not "adventure."

**What To Do:**
Replace CSS cards with pixel art 9-slice frames as detailed in the game-art review. Two frame types: parchment (for information) and iron (for actions). This single change transforms every screen simultaneously because the card pattern is so pervasive.

**Affected Components:** Modal.module.css, HubScreen.module.css (playerCard, goalCard, chestsSection, heroSummary, tile), ForgeScreen.module.css (queue, recipeCard), BarracksScreen.module.css (detail, statCell, equipSlot, enduranceSection), WorldMapScreen.module.css (activeCard, completedCard, regionCard), VillageScreen.module.css (buildingCard), ItemCard.module.css, HeroCard.module.css.

### 2. HIGH: Interactions Lack "Game Feel" -- No Juice

**The Problem:**
Button presses use `:active` with a background color shift (e.g., `rgba(249,115,22,0.25)`). This is a web convention, not a game convention. There is no press-down state, no bounce, no particle burst, no sound hook, no screen shake on important actions. The "Collect" button pulses (good) but collecting an item has no celebratory feedback. Crafting starts with no hammer-strike moment. Level-ups have no fanfare. Selling items has no gold-coin animation.

The chest opening sequence is the one exception -- it has shake, burst, and staggered reveal animations. This is the quality bar the rest of the game needs to meet.

**Specific gaps:**
- **Craft button:** Should have a hammer-strike animation on press. The item appearing should have a sparkle/glow.
- **Level up (hero and player):** Needs a flash, number fly-up, or stat-increase animation.
- **Equip item:** Should show the item "snap" into the equipment slot.
- **Region unlock:** Needs a reveal animation (fog clearing, lock breaking).
- **Sell item:** Gold coins should visually fly toward the resource bar.
- **Send expedition:** Heroes should "march off" or "fade into the distance."

**Why It Matters:**
"Juice" (Game Feel, Steve Swink, 2008) is the difference between a game that feels like clicking spreadsheet cells and one that feels alive. Every successful idle/incremental game from Cookie Clicker to Melvor Idle has satisfying click feedback. The absence of juice makes the game feel like a prototype, regardless of how complete the systems are.

### 3. HIGH: Inventory and Equipment UX is Spreadsheet-Grade

**The Problem:**
The inventory is a vertical list of ItemCards. Each card is a horizontal row with icon, name, rarity label, stats, and an "E" badge for equipped items. This is a data table. It does not look or feel like a game inventory.

Specific issues:
- **No grid layout option.** Players expect to see items in a grid (like Minecraft, Stardew Valley, Diablo). The vertical list makes it hard to scan what you have at a glance.
- **Equipment slots on the Barracks are text labels ("weapon", "armor", "accessory") with inline content.** They should be visual slots -- bordered squares where the item icon sits, with an empty slot graphic when unoccupied.
- **Sorting/filtering controls** (filterBtn, sortBtn) look like form inputs from an admin panel -- tiny grey bordered buttons. They should be integrated into the tab system or use icon-based toggles.
- **The batch sell UI** (batchBar) uses a fixed bottom bar with text buttons. This works but feels utilitarian. The selection mode itself (long-press to enter, dot indicator) is discoverable but un-taught.
- **Item detail modal** piles all actions vertically: repair, dismantle, upgrade, sell. These are competing actions with very different consequences. Destructive actions (sell, dismantle) should be visually separated from constructive ones (upgrade, repair).

**What To Do:**
- Add a grid view toggle for inventory (4-column grid of compact ItemCards showing just the icon + rarity border).
- Redesign equipment slots as visual slot boxes with item icons, not text rows.
- Group modal actions: constructive (upgrade, repair) at top, destructive (sell, dismantle) at bottom with a divider.
- Consider swipe-to-sell or drag-to-equip for faster item management.

## Typography Assessment

**Current:** DM Serif Display (headings) + Outfit (body)

**Issue:** Outfit is a geometric sans-serif designed for SaaS products. It reads as "modern utility" which fights the medieval fantasy theme. Every label, stat, button, and description rendered in Outfit pulls the game toward "dark mode admin panel."

**Recommendation:** Replace Outfit with a font that carries genre weight:
- **For pixel art direction:** Silkscreen or Press Start 2P (pixel fonts that match the target aesthetic)
- **For readable compromise:** Crimson Pro (high-readability serif with medieval character)
- **For stat numbers:** A monospace font (JetBrains Mono or similar) -- the existing `font-variant-numeric: tabular-nums` is correct thinking but Outfit's tabular figures still look like a dashboard

DM Serif Display is a keeper. It works for titles and has the right weight.

## Accessibility Notes

**Good:**
- Color-coded states always have text labels alongside (not color-only)
- `aria-hidden="true"` on sprite SVGs
- Touch targets are reasonably sized (minimum ~36px effective area)
- `disabled` states use opacity + cursor changes

**Needs attention:**
- No `aria-label` on icon-only buttons (settings gear, unequip X, cancel X)
- No focus-visible styles for keyboard navigation
- No reduced-motion media query to disable animations
- Combat replay auto-plays without ability to pause (only skip)
- Lock toast uses only visual notification with no `role="alert"` or `aria-live`

## Desktop/Steam Considerations

The game is locked to `max-width: 430px` centered on the page. On a 1920x1080 Steam window, this means:
- ~75% of the screen is dead black space
- The game looks like a phone emulator, not a desktop game

**Options:**
1. **Keep mobile layout, add frame art.** Surround the 430px game area with a decorative border -- forge workshop illustration, ambient particle effects, resource counters in the margins. This is what many mobile-to-Steam ports do (Slay the Spire mobile, for example).
2. **Add a landscape mode.** Split the screen: left panel for the current screen content, right panel for hero status / inventory / mini-map. This is more work but feels native.
3. **Scale up.** Increase max-width to 800-900px and reflow the layout for wider screens with a media query. Two-column grids where there are currently single columns.

Recommendation: Option 1 is the fastest path and fits the pixel art transformation. A decorative frame around the mobile viewport creates atmosphere and uses the dead space productively.

| Game | Visual Treatment | What F&F Can Learn |
|------|-----------------|-------------------|

| Forager | Pixel art + bouncy feedback | Resource collection should feel this satisfying |

**Current positioning:** Forge & Field's UI quality is at "competent web app" level -- above most game jam prototypes but below any successfully launched Steam indie. The systems design (progression, crafting, expeditions) is ahead of the visual design by a significant margin.
